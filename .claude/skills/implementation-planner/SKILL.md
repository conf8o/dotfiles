# 実装計画スキル

実装計画におけるタスクの整理や優先順位付け、粒度の調整など。

## 実装計画の基本方針

- チケットの情報を把握し、適切な粒度でタスクを分割する
  - チケットに下記情報が含まれていることを確認する。無い場合は不足の旨をユーザーに伝える。
    - 目的・背景
    - 仕様・要件
    - 受け入れ基準
    - 参照実装
- 具体的な実装に寄りすぎてはいけない。また、勝手に具体のまま完成までの実装計画を立ててはいけない。
  - エージェントが仕様を勘違いしたまま進めると、大きな手戻りのリスクがある。ユーザーに確認しながら、段階的に進めることが重要である。
  - そのためにも曖昧な表現によるタスク分解もある程度は必要である。特に、ドメインモデルに関してはユーザーと相談すること。
- ユーザーの把握許容量を意識し、粒度と量を調整することも重要である。具体なタスク量が多い場合、ユーザーへの認知負荷は大きい。
- まずは曖昧な概念でざっくりとタスクを分割し、そのあとでユーザーと相談しながら、必要に応じて粒度を細かくしたり、具体的にしていくスタイルである。

## 粒度決めのポイント

粒度にはいくつかの観点が存在する。ユーザーによる指定のもと、適切な粒度を選択することが重要である。
もしユーザーからの指定がない場合は、根拠とともに候補を挙げつつ提案することが望ましい。

### **PR単位**

一つのチケットに対して、複数のPull Requstを数回繰り返していくスタイル。システムとして動き続ける状態を保ちながら、段階的に実装を進めていく。
重めのチケットを完遂させる際に有効。レビューしやすくすることやメンバーが対応内容を把握しやすくすることが大きな目的。
これにより、レビュー漏れによる要件不足やバグのリスクを減らすことができる。
PR単位ということは、リリース単位ということである。製品として表に出ないものをリリースすることもあるので、デグレに注意しながら少しずつリリースしていく流れとなる。

### **モジュール単位**

モジュールの機能やモジュールごとにタスクを分割する。ある機能について、複数のモジュールを組み合わせたようなタスクに取り掛かる場合などに有効。

### **レイヤー単位**

プレゼンテーション層、ビジネスロジック層、データアクセス層などのレイヤーごとにタスクを分割する。レイヤーごとに異なる技術スタックが必要な場合などに有効。
